## Claude Code 的诞生：一个意外的产品

**Boris Cherny**

还有个快速问题，用 Claude Code 的人，你们主要用 CLI 吗？好的，CLI 为主？Desktop 为主的有多少？VS Code 或 JetBrains IDE 为主的有多少？这其实不太多。其他的？我现在其实主要用 iOS。好的，挺好。

说实话，Claude Code 我一开始其实是无意中做出来的。我在 2024 年底加入这个团队。当时 Anthropic 内部有个类似孵化器的东西，叫 Anthropic Labs。这个团队基本完成了它的使命。我们做了 Claude Code、MCP，还有 desktop app。团队就几个人。所以是一个非常典型的创新团队。我们做了自己想要的东西，然后团队就解散了。现在团队又重新聚起来了，开始第二轮。Mike Krieger——他现在是 Anthropic 的 chief product officer，之前是 Instagram 的创始人之一——现在他在带队做这个。

我当时开始做 coding 的原因，是我们觉得存在一个 product overhang。我猜这里的人经常用这个词，但在我们实验室里这个词确实用得很多。这个想法是说，model 能做的事情远超现有产品能捕捉的。2024 年底我们看 coding 的时候，当时的最先进做法是 type ahead。就是你打开 IDE，按 tab，然后能一行一行地补全。这是 Sonnet 3.5 第一次实现的功能。但我们觉得可以走得更远。model 已经准备好迈出下一步了。所以我们不用再做 type ahead 了，可以直接让 agent 写所有的代码。

所以我就做了，结果前六个月基本没法用。不太好，勉强能用。我当时大概只有 10% 的代码是用它写的。即使后来我们最初发布 Claude Code 的时候，也不是一下子就火了。有很多人用，但没有现在这种指数级增长。真正的转折点是今年五月的 Opus 4。我记得非常清楚。那是指数增长开始的时候，然后每次 model 发布都会再上一个台阶。从 Opus 4，到 4.5，到 4.6，现在到 4.7，一直在持续增长。本质上我们当时在做一个还没有 PMF 的东西，而且我们知道它六个月内都不会有 PMF，因为我们是在为下一个 model 做准备。这个思路贯穿始终。

你知道，Anthropic 一直很专注。我们一直关注 business、enterprise、safety 和 coding。这一直是我们想做的方向。所以某个时候我们知道要做个产品，但不确定具体是什么。结果这个就成了我们的产品赌注。

这个故事太精彩了，尤其它居然是个意外。你公开说过你觉得 coding 已经 solved。如果这是 Anthropic 最核心的三件事之一，能不能告诉我们你具体指什么？还有什么没 solved 的？或者会有哪些次生问题？

**Boris Cherny**

好。我先问另一个问题。这里有多少人 100% 手写代码？有多少人用 agent——比如 Claude Code——写 100% 的代码？有多少人在中间？所以大概 50% solved。对我来说是 100%。Claude Code 的代码库，你们知道的，泄露了，大家都知道。其实很简单，就是 TypeScript 和 React。没什么秘密。没什么复杂的。我们选 TypeScript 和 React 的原因是它们对 model 来说分布很均衡。刚开始建代码库时，model 不像现在这么聪明，所以语言和框架的选择很重要。现在它可以写任何东西，能学没见过的语言和框架。但那时候，你要选一个分布均衡的东西。正因为这样，我很早就达到了让 model 写 100% 代码的状态。这大概是去年十月、十一月左右。

对我来说现在 model 写 100% 的代码。我每天通常提交几十个 PR。上周有一天我做了 150 个 PR。那是纪录。我就是想看看能做到什么程度。对我来说确实解决了。但这不是所有人的情况。有非常大型复杂的代码库。有 model 还不擅长的奇怪语言。但正如大家都知道的，它在进步。通常答案就是等下一个 model。

## 个人 Setup：从手机跑几百个 Agent

**Lauren Reader**

能不能告诉我们你个人的 setup？你前几天给我们展示过，挺惊人的。

**Boris Cherny**

我大概六个月前在 Twitter 上分享过我的个人 setup。有意思的是，我当时没觉得这有什么奇怪——我就是这么写代码的。从那之后又变了。现在我大部分工作其实是用手机做的。你们可能看不清，但我打开 Claude app，左边有个 code tab，我就开了一堆 session。

**Lauren Reader**

有多少个 session？

**Boris Cherny**

通常大概五到十个 session。然后每个 session 里通常有好多 agent，我现在可能同时在跑几百个 agent。每天晚上我通常会有几千个在跑，做一些更深入的工作。管理方式有几种。一种是让 Claude 用一堆 sub-agent 来干活。但我现在越来越多用的是 loop。就是 `/loop`，这东西太酷了。它是最简单但有效的东西。原理就是让 Claude 用 cron 在未来某个时间点调度一个 job，而且是重复执行的。可以每分钟、每五分钟、每天跑，随便你想怎么调度。

我现在有几十个 loop 在跑各种东西。有一个在 babysitting 我的 PR——修 CI、自动 rebase。还有一个保持 CI 健康。如果有 flaky test 或什么的，它会去修。还有一个每隔 30 分钟从 Twitter 抓反馈然后聚类给我看。所以任何时候我都有这么一堆 loop 在跑。我感觉 loop 就是未来。如果你还没试过，强烈推荐。我们还刚发布了 routines，也是同样的东西但跑在服务器上。所以你关掉 laptop，它还在继续跑。

## SaaS Apocalypse？七种 Moat 的演变

**Lauren Reader**

好，我再问一个最喜欢的问题，然后开放给观众。刚才聊了 coding 的变化。我想知道你觉得软件或软件产品的世界里还有什么在变。AI 让写代码的成本降了 10 倍甚至 100 倍，那用软件做出来的产品价值会怎么样？我们会有 SaaS apocalypse 吗？你觉得会怎么演变？又是要做预测了。

**Boris Cherny**

SaaS apocalypse 这个问题我最喜欢了。我觉得会有两件事发生，但都不是大家一直在讨论的那种。第一件事——这里有没有人听 Acquired podcast？对，那是最好的 podcast。我前几周跟他们做了一个 unplugged，感觉像见到我的偶像一样，因为他们的主持人太棒了。他们有个想法叫 seven powers，Hamilton 写了一本书讲这个，是商业里的七种 moat。我觉得因为 AI，有些 moat 会变得更重要，有些会变得更不重要。比如 switching costs 会变得不那么重要，因为你可以用 model 从一个东西移植到另一个东西。还有 process power 会变得不那么重要。对于那些靠 workflow 和 process 做 moat 的公司，Claude 现在很擅长搞清楚 process。尤其是 4.7，它可以 hill climb 任何东西。你给它一个目标，让它迭代直到完成，它就会做。这是第一个这样的 model。

我觉得这些会变得没那么重要，但之前的 moat 还是有用的——network effects、scale economies、cornered resources，这些不会因为 AI 改变。第二件事——如果你看过去十年的 startup 数量，我觉得未来十年能够颠覆一切的 startup 会增加 10 倍。因为现在一个很小的 startup 可以做一个和大公司价值相当的东西，而且可以正面竞争，因为大公司要改变 business process、改变工作方式、重新培训所有人用新技术，会遇到很多内部阻力。但你们这里没有人有这个问题。如果你从零开始，可以从头到底用 AI native 的方式构建。我不知道，我觉得现在是 build 的最好时代，是 startup 的最好时代。会有很多颠覆。

所以我们还是有希望的。谢谢 Boris。现在开放给观众提问，有什么想问的吗？

## 观众问答：Model vs Product 的比重

**Dan**

你说你在 PMF 之前做了六个月，现在 model 已经够好了，你觉得 Claude Code 的成功有多少归功于 model，有多少归功于 product 决策？

**Boris Cherny**

我觉得应该是混合的。大概是混合的。如果一年前问我，比例大概是 50/50。六个月前问，也是 50/50。

**Dan**

那两年后呢？

**Boris Cherny**

两年？我不知道，我们只规划一周。某个月之后的未来吧。顺便说，之所以是 50/50，是因为我以前做过 YC，是某个 YC 公司的第一个 hire，做过很多 startup。在 startup 里，尤其是 YC，他们反复灌输的就是 build something people love。不管产品是什么，不管 model 怎样，最终你还是得做一个人们喜欢的东西。这就是为什么 product 很重要——我们特别在意那些小细节，让你整天用它的时候体验特别好。

随着 model 变得更好，harness 的重要性会下降。我们现在在想的是怎么进化 harness——比如让 loop 成为更一等公民的东西，怎么更容易跑很多 agent。sub-agent 是一个想法。我们还在做很多东西。但我觉得一年后，model 会更加 aligned，所以现在所有的安全机制——prompt injection、静态命令验证、permission mode、human in the loop——都会没那么重要，因为 model 会自己做正确的事。这就是我的预测。谢谢。

## 软件民主化：类比印刷机革命

**Audience Member**

把视角拉大到软件之外，我觉得 Claude Code 几个月前做了一次文化上的改变，它让构建软件民主化了。你可以看到店主给自己做软件，甚至编程 microcontroller 来控制开门时的灯光。你觉得未来构建软件会不会变成像懂 Microsoft Office 一样的技能？是人人都能做的事，不只是科技行业的人？

**Boris Cherny**

天哪，是的。绝对是的。我觉得会更进一步。我觉得会变成像发短信一样的技能。我平时主要读两种东西：科幻和技术史。这是我看最多的。技术史里有一件事，我觉得是现在发生的事情最清晰的类比。就是 1400 年代欧洲的印刷机。

印刷机出现之前，欧洲大概只有 10% 的人识字。他们知道怎么读写。他们经常被不识字的国王和 lords 雇佣。他们的工作就是读写，这不是人人都会的事。印刷机发明后，然后又有两台，在第一台印刷机之后 50 年里，欧洲出版的文献比之前一千年加起来还多。同期书的成本降了 100 倍。然后花了大概几百年——因为学读写很难，需要教育系统、政府，大家不能都在农场干活——但接下来的几百年里，全球识字率涨到了 70%。现在人人都能读写，不需要读写的学位也能读写。当然还是有专业作家，那也是一个职业。

我觉得接下来会发生的事情——会比 50 年快得多——是软件会变成完全民主化的东西，人人都能做。这有很多推论。比如假设你要写会计软件。我觉得最好的会计软件作者可能不是 engineer，而是很好的 accountant，因为他们懂领域，coding 是简单的部分。懂领域才是难的部分。我觉得这就是显然的未来。

## Anthropic 和世界的差距

**Audience Member**

你们算是活在未来了，因为你们能用到 model 和 agent。Claude Code 发布前是内部工具。你们 engineering 的水平和世界其他地方的差距，是一个月？三个月？六个月？这个差距是变大还是变小？

**Boris Cherny**

对，内部我们用的 model 和大家一样。对我们来说 dogfooding 非常重要。我们用和大家一样的东西。我们用一点 mythos 来试，然后用很多 Opus 4.7 来 dogfood 和写大部分代码。我觉得 model 层面没有差距。基本上就是 mythos，某个时候某个版本会开放给大家。我觉得 product 层面差距可能更大。这和我们改变所有 process 有关。如果你问 Anthropic 的人，我们用 Claude 做所有事情。我们的 Claude 整天在交流——我 coding 的时候，我的 Claude 在 loop 里跑，它们会通过 Slack 和其他人的 Claude 交流——那些也在 loop 里跑——来搞清楚未知的东西。公司里没有手写的代码了。所有 SQL 都是 model 写的。所有东西都是 model 构建的。

所以我觉得我们领先的其实不是技术——因为同样的技术大家都能用——我们本质上是在建平台，所以对我们很重要的是开发者能用我们用的东西，我们 dogfood 所有发布的东西。但我觉得更大的差距其实在于组织结构和组织 process。这是希望能通过像这样的场合来交流，让大家也能学习和进化。我觉得这也是 startup 的优势。从头开始这样做会容易得多。

## Multi-Agent 和并行化

**Jared**

上次我们聊的时候，我记得你提到 multi-agent，当时还在代码层面——在之前的 Sequoia 活动——你说有一些东西在做，一些你在想的。现在显然有 `/batch`、`/loop`、sub-team、team。能不能说说，在 model 层面和 harness 层面，你怎么在 harness 里注入 prior，model 的 objective function 怎么变化，来让 delegate work、spin up agent 的体验更好？因为很多工作是可并行化的，可以更快做更多事，但我感觉我得自己叠加直觉来判断什么时候该并行化，而不是 model 自己理解可以 spin up 10 个 sub-agent。

**Boris Cherny**

Product 层面其实就是 prompting。就是这样。我们会调整 prompt 来帮助 model 更多地并行做事。但说实话，随着 model 变好，它自然而然就会这么做。像 loop，我发现 4.7 开始自动做这个了。很酷。比如我告诉它"去拉这个数据查询"，它会说"嘿，我发现数据在变化，我启动一个 loop，每 30 分钟给你报告"。我就说"好，能发到 Slack 吗？"然后它用 Slack MCP 来做。所以我觉得其实时间推移后，不是用户要去想怎么更好地用工具，如果还是那样的话，就是产品设计有问题，是我没做好。应该是 model 更擅长这些，以及我们在 prompt 上引导它自然地这么做。

## 云端 vs 本地 Agent

**Audience Member**

我们现在用 Claude 或 Codex 或这些云端工具做很多计算。但也有一些很 vocal 的 advocate 提倡本地 AI。我能想象随着 open-weight model 和其他东西跟上，这可能会成为更多人的一种选择，获得高质量 coding assistance。我想知道你对接下来几年的愿景。你觉得轨迹是大家继续依赖云端集中计算，还是会转向我们都有自己的本地 agent、不会被 throttle 等好处？

**Boris Cherny**

我觉得有几种回答方式。最根本的回答方式是：不重要。因为我们现在到了这个点：model 自己能搞清楚。我觉得几年后，model 会做所有代码，会启动 agent，会构建环境。如果它决定"其实我会用本地 model 来做这个"，那它就会这么做。我觉得这些决定不再是我们作为 engineer 来做的了。

## Co-work 和 MCP：知识工作的未来

**Jamie**

Claude Code 的一大决策是利用了开发者工具和 workflow 很多都在本地这个事实。但对于一般的知识工作，不一定总是这样，有云端工具等等。我想知道你们在 Co-work 上怎么思考这个问题——怎么给 Co-work 足够的工具访问权限，让它像 Claude Code 对开发者那样强大？

**Boris Cherny**

好问题。我在大公司的时候，我们花了五年把所有环境迁到 remote。工作量很大，尤其是大规模的时候。但知识工作其实已经大部分在云端了——Salesforce、Docs 等等。对我们来说，答案很简单——就是 MCP。你在 Claude AI 里用的那个 MCP connector，接上 Salesforce、Google Docs、Google Calendar。然后 Co-work 能用它，Claude CLI 能用它，Claude Code 任何地方都能用它。

**Jamie**

对于没有 MCP 的东西，你觉得 computer use 会是一个大机会吗？

**Boris Cherny**

我觉得 computer use 是一个 catchall。目前据我所知，Anthropic 在 computer 上领先挺多的。如果你通过 Co-work 用，效果挺好的。它能用你电脑上的几乎所有软件。很慢，但做得挺好，尤其是 4.7。但我觉得其他情况下 MCP 是答案。而且这些东西没那么重要——可以是 MCP、API、某种程序化访问，因为 model 不在意。对 model 来说，就是 token。
