## Lab 平台与 environment 的概念

**Johannes Hagemann**

我们有一个全栈研究平台叫 Lab。Lab 的目标是让每个人都有能力做前沿研究实验室内部能做的事情。Lab 的一个核心重点就是 environment 这个概念。RL environment 封装了做 reinforcement learning 所需的一切，让模型可以通过试错来改进。但它不止于 RL。如果你以前没听说过 environment，它基本上就跟人们谈论新模型发布时报告的 evals 是一回事。SWE-bench、AMIE、Terminal Bench 都是 environment 的例子——它们有一组任务数据集，有一个供模型使用的 harness，还有一个负责对输出质量评分的 rubric 或 reward function。你在离线评估中用作 test set 的东西，完全可以在 reinforcement learning 中用作 training set。

我们的平台让用户能在 workflow 中使用 environments 来做 post-training、evaluation、synthetic data、reinforcement learning。它也是一个像 GitHub 一样的社区平台——这些东西是新的、复杂的，我们希望人们有大量可用的构建模块和示例，也希望大家可以协作、分享。

## 产品-模型优化循环：Cursor 的启示

## Environments 就是 Evals

**Training Data Host**

我想追问一下你说 environments 其实就是 evals 这个观点。在我理解里，environment 是 state，是对世界状态的描述——你观察，采取行动，观察世界状态的变化，然后更新 world model。这在我看来和 eval 是不同的。你能帮我弥合一下这两个概念吗？

**Johannes Hagemann**

在一两年前，eval 更多是问答形式的——一大题库。而 environment 的概念，人们想到的是老式的 RL，比如 Atari——那种长时间的状态交互循环。但现在两者已经合二为一了。特别是那些你想用来做大规模训练的 environments，它们有复杂的状态，可能是在模拟一个 web app，是一个完整的编码平台。但在原始的 RL 游戏中一直有一个 reward、一个 goal，有办法衡量进展和性能。这就是我所说的——存在一个目标，有办法衡量表现好坏。

## Harness 与 Environment 的关系

**Johannes Hagemann**

Harness 是 environment 的一部分。对于任何 eval 或 environment 任务，有输入，一系列操作，然后有输出状态被评分。这整个中间的部分就是 environment，而 harness 定义了模型如何与系统的其他部分交互。对于 coding agents，CLI coding agent 是 harness，terminal 是 environment。但有些情况下 system prompt 加一些工具就是 harness；有些情况下会产生子 agent，子 agent 也有自己的 harness。我们用 environment 这个抽象概念，因为 agent 太窄，harness 也太窄——但 environment 作为一个整体抽象，任何系统-模型交互都在范围内。

## 什么时候应该构建 RL Environment

**Johannes Hagemann**

我觉得两者都有适用场景。如果你要训练一个非常擅长写 Rust 代码的模型，你可能希望它对很多不同应用场景都好用，所以你会用不同的 environments。还有一种情况是产品本身就是面向终端用户的界面，用户直接和 agent 交互，这种情况下最合理的做法可能就是让产品直接变成 environment。那些在意用 Claude 还是 GPT-5 还是 Gemini、在意能否选择模型的公司，它们已经在构建的基础设施和做 RL 所需的基础设施是同一套。所以 RL 是一个非常有用的工具，让人们在系统优化工具箱里多一个选项。

Environments 可以做更多事情而不只是 RL——比如从大模型蒸馏：把大模型放进 environment 运行很多次，然后用 grader 过滤出最好的例子做 SFT fine-tuning。也可以用 environment 做 prompt optimization、不同模型的 AB test。每个 AI 公司都应该优化他们的 AI 系统。

## 网络安全环境与 Sim-to-Real

**Training Data Host**

我听说有些基础模型公司花了几百万美元在每个 environments 上。构建一个网络安全 environment 需要什么？人们构建的东西能真正扩展并反映现实世界的复杂性吗？

**Johannes Hagemann**

原则上它们可以做到你想要的任何复杂度。任何在计算机或计算机网络上的东西都可以成为 environment。真正的瓶颈是模拟器的成本。很多精力花在找巧妙的方法来 mock 系统的正确部分。比如你想让 agent 搜索整个互联网，有时需要真正的网络搜索，有时可以设计不需要完整搜索引擎的任务。有一个 benchmark 叫 Tattoo Bench，涉及客服 agent 和数据库——在真实系统中你需要完整数据库，但如果你知道 agent 只会问特定类型的问题，你就可以创建一个更便宜的内存 mock 数据库。

我们围绕这一块做了很多工作。以 coding agents 为例——sandbox、terminal states、快照功能、与不同 agent harness 交互的协议——这些都带来了大量复杂性。我们在设计时尽量保持开放，留出空间来构建新功能。从基础层的通用 environments 开始，你可以继续深入：coding agent environment 带全局 sandbox，或者每个 rollout 一个 sandbox。我们也花了大量心思在开发者体验上——文档、agent 的 skill files。

## 为什么要创建 Environment Hub

**Johannes Hagemann**

环境中心的想法源于我们看到很多开源仓库里有各种环境的混乱实现。在推出环境中心之前我们已有一个很好的验证器框架。除了共享环境、拥有一个开源平台之外，关键还在于有一个地方可以围绕它们构建大量基础设施，这是上传到 GitHub 做不到的。拥有适当的评估系统与环境集成，可以立即在所有前沿模型上测试，这是人们大量使用环境中心的原因之一。

**Training Data Host**

社区行为是什么样的？你看到有人在复刻、修改这些环境吗？

**Johannes Hagemann**

确实有很多合作者希望保持私有，这可以理解。但拥有中心的优势在于——可以在已知环境上做消融实验，将自己的私有环境与公开的同类任务环境对比。对于评估来说，有统一实现的流行基准测试也很有价值。最流行的环境往往是文档中的示例——Wordle 和 Wiki Search 是受欢迎的例子。Wiki Search 是在维基百科上做搜索，但被设计成可更广泛用于 agentic 搜索的模板。

## RL 的效率问题与上下文限制

**Training Data Host**

Andrej Karpathy 也是你们的投资人之一。他有一句名言：RL 很棒但效率很低，就像用吸管吸数据流。你们同意吗？

**Johannes Hagemann**

RL 确实用了大量计算获取相对较小的信息信号，这是事实。但某种程度上这也是它的价值所在。很多实验室聚焦 RL 是因为人类数据是难以扩展的瓶颈，尤其是高质量的人类数据。RL 允许你用计算换取数据——用更多计算从更少数据中获取大量价值。没有 golden examples 时它也很有效。也许人们会发现比 RL 更高效的探索方法，但这就是目前用计算来探索和提升能力的前沿。

看看 Claude Code——它之前已经很受欢迎，但在过去一个月左右更加火爆了。总的来说，我们认为这不会是终点。在推动 agentic AI 能力方面还有很多低挂的果实。但我们也很清楚目前的局限性——就是上下文方面。目前可塞进一个上下文窗口的 token 数量有限，我们一直在思考如何改进这一点。

## 开放权重与封闭权重模型

**Training Data Host**

你们的基础设施是在开放权重模型上最优工作，还是也能帮助人们在封闭权重模型上做 post-training？

**Johannes Hagemann**

训练器本身需要访问权重。但 environment 的概念是通用的，我们可以在 environment 层面用相同的基础设施对封闭模型做 eval、prompt tuning、模型选择、评估 agent harness。无论是使用开放还是封闭模型，或者把 environment 当作一个数据引擎，有很多方法可以用这些工具来优化模型。LoRA 属于 fine-tuning 过程的一部分，你不需要完整权重。但在很多情况下，你需要 LoRA adapter 还是完整 fine-tuning，取决于训练方案和优化目标。

## Recursive Language Models 与长 Horizon Agent

**Johannes Hagemann**

我们对长 horizon agent 很感兴趣。我们内部已思考了很久如何让模型学会管理自己的上下文。Recursive language model 的研究方向是我们的重点之一。从 Alex 发表 RLM 工作以来我们就一直非常感兴趣。几周前有博客文章展示了使用 RLM harness，给语言模型访问一个持久 Python 包装器中的变量，让它检索、转换和自行管理上下文。这被证明在长 horizon reasoning 基准测试上有显著提升。我们非常兴奋——真正训练模型使用这个 harness，将是接下来几个月的工作方向。

## Synthetic Data 与 Continual Learning

**Johannes Hagemann**

我对 synthetic data 研究非常兴奋。感觉有很多事我们理论上应该能做，但还没在公开领域看到——比如自我反思的创造性方式。大家经常讨论 continual learning，模型自我学习的能力会越来越重要。将已知的技巧以不同方式组合——prompt optimization、distillation、synthetic data——感觉还有大量探索空间。让模型策划自己的训练数据，甚至策划自己的 environments，以及理解哪些版本对终身学习最有效。
