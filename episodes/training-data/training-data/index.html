Cursor 如何在 Fireworks 上训练 Composer：高性能强化学习的分布式基础设施

本期节目深入探讨 Cursor 如何与 Fireworks 合作完成 Composer 2 模型的大规模强化学习训练。Federico 和 Dima分享了为什么 Cursor 决定从应用公司转向自训练模型，以及他们如何通过持续预训练加强化学习两条轴线打造高性能编码模型。讨论涵盖了分布式训练基础设施的设计难题，包括异步流水线、全球分布推理集群、权重增量压缩传输、MoE 模型的数值对齐挑战、Router Replay 技术，以及长跨度智能体的上下文自我总结机制。两位嘉宾还探讨了实时强化学习与模拟 rollout 的权衡、LLM 作为评判者的角色、以及强化学习环境应该尽可能接近生产环境的理念。

为什么 Cursor 要自训练模型

Host

今天我很高兴邀请到 Cursor 的 Federico 和 Fireworks 的 Dima 来参加播客。Federico，你是 Cursor 新的智能体编码模型 Composer 2 的研究负责人。Dima，你在过去几个月花了多少时间兼职支持 Cursor，来完成这个庞大的训练任务所需的基础设施建设？我很兴奋今天能和你们两位谈谈 Composer 2 的训练是如何完成的，你们一起解决了哪些难题，以及你们认为这对 AI 和基础模型公司的未来意味着什么。

Federico

很兴奋，谢谢邀请我们。

Dima

谢谢邀请。

Host

我们直接进入正题。对于那些没有密切关注的人来说，Cursor 最近宣布了 Composer 2，这是一个用于长跨度编码任务的智能体编码模型。Federico，直到现在，Cursor 大部分时间都是在让其他人的编码智能体运行起来。Cursor 为什么要如此大力投入 Composer 2？从应用公司变成基础模型公司，这对你们来说有多核心？

Federico

我们开始研究训练自己的模型，原因可以这样理解：模型就像一个存储盘，它的权重里可以存储一定数量的比特信息。这个想法很简单，我们只关心一个任务。我们甚至不关心编码或编程本身，我们只关心 Cursor 内部的软件工程，而且只在 Cursor 内部。那么，如果我们把模型权重能够存储的所有信息比特都分配给这一个特定任务呢？

另外，大家可能已经注意到，Composer 比 Opus 和其他编码模型便宜一个数量级，因为我们可以简单地把所有模型权重专门化到那个特定任务上。所以我们可以用一个更小的模型来服务，诸如此类。

Host

所以核心思路是确保我们拥有的每一个权重比特都专门用于我们手头的具体问题。

Federico

没错。

Host

这看起来是一个几乎可以推广的问题。Dima，我好奇你的看法。你认为每一家应用公司都应该把 Cursor 看作未来的风向标吗？他们都应该考虑做同样的事情吗？

Dima

绝对是的。我们实际上把这看作应用演进的一个模式。你可能开始做原型，用现成的模型让东西跑起来，做一些提示工程，搞清楚你的 harness 怎么工作。但你应用里最有杠杆效应的属性是用户数据的使用，或者你应用特有的具体方面，比如你的 harness 的某些方面，你提供哪些工具，应用怎么工作，这些都是对你的应用很重要的关键信息。

捕捉这些的正确方式，你可以通过提示做到一小部分，但真正的正确方式是定制你的模型让它能在你的环境里行动。

Federico

绝对是的。智能体调用的某些工具，很难简明地向模型描述那个工具的确切行为。通过后训练，我们可以把使用这些工具的最优方式固化进去。比如 Composer，我们确实会给 Composer 发送一个提示，但我认为我们训练它的方式，即使没有提示它也能工作，它知道该做什么，因为我们本质上就在训练过程中把模型推向正确的方向。

Dima

提示工程的上限基本上就是这样。如果你想打造真正伟大的 AI 产品，你必须通过微调来影响模型行为。这是第一个原因。第二个原因是 Federico 提到的成本权衡或性价比权衡。在 Fireworks 我们这样看：当你试图做优化时，你有一个质量、速度、成本的三维权衡。

你可以通过仅仅优化基础设施走得很远，我们最初和所有客户都是这样做的。但当你开始进入模型训练，你可以把这个权衡推进得更远，可以用更低的成本获得更好的模型，运行得更快。Composer 就是一个很好的例子。

关于 Bitter Lesson 的讨论

Host

我想再追问一下。我想问问这个方法是否违背了 Bitter Lesson。我们进来之前其实都在聊 TabNine。我记得在 LLM 时代之前，有这些小型专用编码模型。我觉得让很多人惊讶的是，当你把模型做大，只是用互联网上大量的英文文本和其他语言训练，实际上模型本身在编码上也变得更好了。所以至少我看到的趋势是：更大的模型在所有事情上都表现更好，包括编码。你们说的这些，是否和 Bitter Lesson 的思路相悖？

Federico

我觉得不违背，但有一点要指出：实验室训练的大模型也训练了大量代码。代码是实验室关注推进的主要任务之一，所以它们不只是泛化到代码，它们也做了专门化。对于我们的情况，如果你相信 Bitter Lesson，我们其实是在数据维度上大力推进，我们知道模型本质上容量有限。所以如果要饱和所有容量，我们需要扩展数据。为了摄入更多数据，我们需要把权重从模型可能有的干扰中解放出来。

Composer 2 的训练方法

Host

超级有意思。我们深入聊聊 Composer 2 的训练。你们几周前发布，立刻引起关注。基准测试数字很强，推理成本很低。简单说说 Composer 2 怎么工作，你们做了什么让它这么高效？

Federico

我们从一个非常强的基座开始，就是 Kimi 2.5。这是一个 1 万亿参数的 MoE（混合专家）模型，激活 30B，所以实际上非常稀疏。我们看了一下整个局面，发现有两条轴线。Composer 1 主要只推其中一条，就是强化学习，但 Composer 2 在两条不同的轴线上推进：一条是持续预训练，另一条是强化学习。Composer 2 的优势就在于在这两个方向上都推进了。

我们开始训练时，先在代码 token 上做了大量的中间训练，规模几乎是预训练级别的。然后从中间训练的检查点出发，我们做了大规模的强化学习，在很多很多任务上训练。

Host

这里的前提是，因为 Cursor 处在大量有趣的编码 token 的中间，你们实际上相当独特地拥有接近预训练规模训练所需的数据。

Federico

是的。

Host

那为什么不自己预训练一个模型呢？

Federico

我们是从上往下思考，而不是从下往上。怎么在最短时间内给用户一个有用的模型？如果我们从底层开始，先搞清楚怎么做预训练，然后扩展到中间训练，然后好了现在搞清楚了中间训练，再做强化学习——那要花很长时间才能给用户一个模型。反过来做，我们能在很短时间给用户一个有用的模型。

所以希望下一版 Composer 会是我们自己的模型，而不是基于开源基座。

Host

模型在中间训练阶段大概学什么？在后训练阶段学什么？

Federico

中间训练阶段，它大概是在学习代码库，学习很常见的特定代码模式，还有世界知识，里面也有网页数据。这基本上是在创建一个更宽的分布，然后强化学习可以在此基础上锐化。在强化学习阶段，模型可以直接和 Cursor 的 harness 交互，学习它余生将要生活的世界。然后强化学习阶段，它学会正确调用工具，如何导航环境，如何写出正确的代码。

因为中间训练阶段，它学会写代码，但不一定学会写正确的代码。我们尽量只训练正确的代码，但模型实际上不知道怎么区分这两者。而在强化学习里，我们做的一个关键事情就是调整模型的特性，告诉它：嘿，现在你得一直写正确的代码。

Host

有意思。中间训练后的模型，和你們 Tab 自动补全的模型类似吗，还是不同的核心能力？

Federico

我想可以这样说，因为中间训练我们就是做下一个 token 预测，预测下一个 token 再下一个。

Host

那为什么不直接预训练你的 Tab 补全模型呢？为什么中间训练是一个不同的模型？

Federico

Tab 是一个非常小的模型，因为它需要超低延迟。是的，我们希望它非常快。所以这两个基座模型的核心区别是：Tab 是小的，Composer 相当大。

大规模强化学习的基础设施挑战

Host

看来你们为 Composer 2 做的主要工作是这个大规模强化学习训练。能拆解一下吗？这里面有什么，你们解决了哪些难题？

Dima

做 rollout 和预训练、中间训练很不同，因为预训练和中间训练你只是预测下一个 token，而 rollout 你实际上在运行整个 harness，整个实验。你让模型在环境里行动，看它在给定 rollout 里表现如何——这就是所谓的 rollout——然后根据它是否正确做了某件事给它奖励，可能用 LLM 作为评判者，或者用可验证的东西比如代码是否编译等等。

这意味着相比常规训练，你需要一堆其他组件。你仍然需要大规模训练，需要调度数万个 GPU 做前向反向传播，做所有中间训练和预训练做的事。但你现在还需要调度一堆环境，需要运行模型推理——因为当你做这个 rollout，你实际上是在运行某种意义上的真实 Cursor 会话，对吧？

Host

这是一个前向 pass 吗？

Dima

不，rollout 基本上是你的整个智能体会话。可能需要 50 个回合。模型拿到初始提示，决定调用一些工具，你执行那些工具，模型生成一堆其他代码，这整个会话就是你跟 Cursor 智能体交互的过程。你把这个完整会话模拟作为训练的一部分。你得到最终奖励，用那个信号回到训练器，把它整合进模型权重。

所以你有一个非常大的更新循环，非常异构，因为有所有这些不同组件一起工作。现在你要调度所有这些高效工作，高吞吐量，因为 GPU 很贵，你想快速经济地训练模型。这是一个算法和基础设施交叉的有趣问题，因为有很多权衡，你可以怎么协同优化和协同设计系统。

一个方面是所谓的异步并行或流水线并行。基本思路是：你在步进更新模型，有当前模型版本，要用它做一堆 rollout。那做 rollout 的时候训练器干什么？朴素的方法是：好，现在暂停训练器，做一堆会话，那些会话可能跑 5-10 分钟甚至更长，拿到结果，暂停推理，回到训练做更新。这理论上算法上很稳健，因为你精确模拟一切，但系统上非常低效，因为一半容量一直闲置。

你可以用聪明的算法技巧把这一切流水线化。想象成一个巨大的工厂，有训练器大楼，有 rollout 大楼。它们一直在运转。rollout 总是拿最新模型版本，做新会话，模拟新智能体会话，训练器总是拿新结果来，计算更新。所以一切一直在动。

权衡在于——我说算法上不同是因为，当你完成某个测试 rollout，模型权重可能已经用其他数据更新过了。所以有这种陈旧性，延迟，模型能学习更新有多快，因为当你处理完某个交互会话，模型权重变了，这引入了有趣的训练动态，有聪明的办法解决。但反面是，所有 GPU、所有计算机都在负载运转，这实际上让你用更多 flops，回到你的 Bitter Lesson 例子，你有更高的计算效率，可以在更短时间得到更好的模型。

你可能因为异步、不做完美数学更新损失几个百分点，但你通过不让一半容量闲置大大补偿回来了。这里面有很多深度有趣的交互。

Federico

Cursor 对性能非常认真，因为和大实验室不同，我们有几千个 GPU，不是几百万个。所以我们用各种技巧榨干 GPU，我们生产训练甚至用 FP4。我们和 Fireworks 合作推进推理，因为强化学习基础设施本质上比预训练更复杂——你需要所有预训练基础设施，这只是其中一个要求。然后你需要所有运行这些环境的基础设施，必须尽可能接近用户的电脑，这非常重要，因为有时候模型能发现它是在假环境还是真实环境里运行，它在强化学习时的行为和生产环境不同。

Host

你是说它能意识到自己在假环境里，开始行为不同？

Federico

是的。是的。有意思。就像：哦，我在假环境里。我学了几招在这个环境里拿更好奖励，让我试试。模型喜欢作弊。强化学习真的很擅长鼓励作弊。

然后我们需要非常高效的推理。这很重要。有一个迷思说强化学习时推理 flops 远多于训练 flops，这其实只是因为开源推理引擎非常不优化，而不是强化学习的固有属性。理论上比例差不多。如果你把 GPU 推到极限，应该有三分之一训练 GPU 分配给推理，因为训练实际上是三个前向 pass——前向 pass、数据梯度、权重梯度。如果你在推理上真正达到临界批次大小，应该只有一个前向 pass 的 flops。

Host

这就是你们用 Fireworks 而不是用开源推理引擎的原因？

Federico

另一个选择是我们自己建一个，但我们的工程师有限，和所有人一样。我们宁愿让工程师把训练做得更高效更精确，而不是启动一个推理工程。

全球分布式训练

Host

你们技术论文里提到你们用全球分布式方式做这件事。为什么全球分布式，这有什么难点？

Federico

有各种原因。一个，这种非常大的连续集群在市场上很难找。我们可以用一个集群跑所有训练——我们不能做全球训练集群——但强化学习的推理部分，我们可以全球分布到世界各地的小集群上。Composer 2 训练我们用了四个集群，分布在世界各地，彼此很远。

我们甚至在生产流量低谷时用了一些生产流量。Composer 1.5，上一版模型在服务，当用户用得最少的时候，我们就抓一些推理 GPU，用它们加速训练。我们可以做这些事，不用一个大的连续集群就能轻松扩展训练规模。

Dima

强化学习训练非常异构，通过利用异构性——不同组件需要什么基础设施——你可以驱动效率。这个模式到处都能看到。具体到训练，你有所有这些高度互联的集群，需要高速网络，需要同步工作，所以这些集群很贵。实际上很难找到大的，在 Composer 训练的规模上，找两倍大的集群比找当前大小的难得多。

所以如果你能把组件拆分放到不同地方，第一，你不需要找那么大的集群。第二，你可以找不同硬件的权衡，因为推理不需要那种宽互联。你可以有更小的 GPU 组互联在一起。可以有异构的 GPU 组。可以有不同代的 GPU。你可以玩所有这些优化游戏。最后，推理更容易上下扩展。

当你有非高峰时段，你可以把所有推理池看作一组 GPU，服务真实用户的生产流量，或者服务强化学习的模拟环境，在两者之间平衡。当然，这是一个很有趣的系统问题。

Federico 提到 Kimi 模型是 1TB。训练一步大概 5 到 15 分钟。这意味着每 5 到 10 分钟你就产生 1TB 的新权重快照。问题是：你怎么高效地把它送到世界另一端的集群？你想快送，因为记得你不想让陈旧性失控。

我们一起解决的最有趣的问题之一是：尽管完整模型是 1TB，并不是所有权重每一步都变化。强化学习做很多非常精确的调整。特别是在训练进行时，哪些权重子集变化有很规律的 pattern。如果你看一步训练后模型变化多少——10 分钟后——那之间的 delta 相对很小。

你可以写一个压缩算法利用这个属性，现在你遇到的是数据库系统的问题：我有 delta，我想把它送到世界另一端。我的 delta 可能比送完整模型小 20 倍，这就变得可行了。但当然你现在需要建所有这些机制——完整快照和 delta 的存储系统、恢复、协调等等。我们用无损方式建了，意味着你在另一端总是得到比特等价的模型，不用担心任何数学方面的问题，而且可以很快。

最坏情况下几分钟内能完成，通常不到一分钟，最重要的是你只暂停大概 30 秒来在你的实际推理模型里切换权重。我们还通过分片上传和下载完全饱和了集群的带宽出口。你可以用所有这些系统技巧把延迟降下来。确实有不少复杂度，但你可以把它抽象出来，让它工作得很好。不干扰训练算法，反面你有这种能力拆分，利用其他集群来做。

这某种程度上违背了强化学习基础设施应该怎么做的传统智慧。传统智慧是：你会有一个很大的集群，用 RDMA 连接，非常贵，可能分配三分之一训练、三分之二推理。当然，如果你有非常贵的网络，复制这 1TB 很容易，但现在你有三倍大的集群。如果你的推理引擎更优化，可能你在 GPU 上省三分之一，然后把一半集群放到别处，可能是更便宜的硬件，不同区域。所以成本降了不少。

Host

你们描述这个的时候都在笑，因为这太难了，这是系统工程师的梦想。你们建的系统太棒了。

Federico

很多晚上都在做这个。

Host

你们看起来花了很长时间在一起。

Dima

很多时间在一起。

MoE 模型的数值对齐难题

Host

你开头提到 Kimi 是一个非常大的稀疏 MoE 模型。这让强化学习训练有什么棘手之处吗？

Federico

是的。

Host

怎么棘手？

Federico

做推理时，你实际上做的是一个前向 pass，就是自回归的。这个前向 pass 产生采样 token 的 log 概率。当我们把模型的生成送回训练器时，我们必须重跑那个前向 pass，因为我们做异步训练，所以产生那个 pass 的模型可能实际上比训练器落后几步。我们要重跑前向 pass，复现 log 概率。

问题是理论上，如果是同一模型版本，这些 log 概率应该完全一样。但即使是同一模型版本，同样 token 的 log 概率值会略有甚至很大不同。这通常叫做推理的数值不匹配。这些天你到处能听到这个。

Host

为什么？为什么会发生？

Dima

主要是因为本质上浮点运算是非确定性的。你在代码里学到的是 A+B+C 和 C+B+A 结果一样。用整数、用整数做，这总是成立的。但用浮点数——实际上是近似数，有尾数和指数等等——A+B+C 和 C+B+A 会给你不同结果，甚至 A+B 和 B+A 也不同。

基本上本质上是所有运算的累加顺序——模型做的基本上是乘法和加法——加法顺序影响最终结果。都是小差异，但经过百万亿次运算会放大。所以你做模型推理时，通常没那么重要，因为你预训练的模型实际上很鲁棒。如果你翻转一些比特，它仍然产生好结果，你的基准测试不会变。

但强化学习特别重要，因为你用非常弱的信号教模型，这些数值差异的噪声可能让训练成败。这特别重要。又是算法和系统部分的有趣交叉，因为你可以写漂亮的数学，但实践中不工作。

有办法把这个差异几乎降到零。有所有这些批次不变的方式，你可以非常非常小心，写所有 GPU kernel 让它们总是用同样顺序加数，总是 A+B+C 而不是不同顺序。这是可能的，但总是有权衡。你的系统可能变慢 2 到 3 倍。

又变成一个有趣的权衡：有没有 10% 的减速我们可以接受，或者实际上大概几个百分点我们可以接受来解决 90% 的差异？这是我们通过迭代一起找到的正确权衡。

Host

你提到这对 MoE 和稀疏性特别难。原因是什么？

Dima

MoE 工作的方式是：你每层拿激活，跑过一个 gating 层，基本决定：对于这个 token，我要从 384 个专家里跑这 8 个。它会做一些计算，比如 top eight 分数。那 8 个专家被激活，其他不激活。这个运算把小的数值差异放大很多，因为你的隐藏状态可能只差小数点后第五位，不重要，但这个差异让你选了专家 7 而不是专家 9 作为 cutoff，突然你激活了完全不同的模型部分，差异放大了很多。

MoE 模型天生对这种不匹配更敏感。做推理或常规 rollout 时通常不重要，会平均掉，但如果你要让这个模型学习，这差异巨大，因为你的推理激活了专家 7，但训练时你要更新专家 9，它推理时甚至没贡献。

Host

所以你们手写 GPU kernel 来解决这个问题？

Dima

是的。你可以通过 GPU kernel 解决所有这些，总是有权衡。特别是 MoE，你可以做一个有趣的技巧，人们叫 router replay，但基本上你可以让你的推理把额外信息传给训练，说：嘿，我对这个 token 激活了专家 7。这很小一块信息，就是一个整数，说这是我激活的专家，训练器可以对齐。

很多数值对齐就是做这种技巧，匹配量化级别，匹配 kernel 等等，把训练和推理实现的分歧降下来。这造成巨大差别，要么你完全发散，要么你需要更多数据来解决这个不匹配，计算效率更低。

长跨度智能体与上下文管理

Host

你提到 rollout 每次几分钟。看起来整个领域都在推向长跨度智能体，能长时间不间断工作，一般不失败。你们做了什么让智能体能跑更长？

Federico

好几件事。强化学习的一个问题是轨迹越长，信用分配越难。想象我们在底线给 thumbs up thumbs down——在它工作结束时。简化说，模型问自己：好，哪里做对了，哪里做错了？这就是信用分配问题。越长越难，你得用一些技巧。

另一个问题是你空间不够了。这些模型有有限的上下文窗口，到某点会达到。Cursor 解决这个的方式是我们把 compaction 放进强化学习循环里。我们叫自我总结。在强化学习期间，智能体实际上学会如何继续，永远走下去。

实际上我们的模型是 20 万上下文窗口模型，但实际上它可以跑到百万 token，因为这种能力：它可以总结工作，然后拿那个总结重启上下文窗口，同时仍然在完成任务。通过强化学习——强化学习把模型推向正确做事朝目标前进——同时我们在训练模型产出好总结，然后训练模型很好地听那个总结。这有点像是推理的延续。

Dima

这很迷人，因为通常上下文管理被认为是 harness 的一部分。这种情况下，你实际上协同优化 harness 和模型本身，把所有这些扔进优化循环。我们在 AI 里一次次看到这个：你把越多计算扔向问题，你能端到端解决越多问题。计算魔力和 Bitter Lesson 生效，你得到能更好一起工作的系统。

Host

你认为每个公司都会强化学习训练他们自己的 harness 吗？每个公司都有 Cursor 这种形状的问题吗？

Federico

如果他们在用 AI，产出很多 token，有产品可以优化，我觉得是正确方向。

Host

有意思。看来你们做的强化学习大部分是在 harness/工具调用部分，而不是在让模型擅长预测代码下一个 token。这是其他创始人考虑哪里该用强化学习的模式吗？如果你想让智能体用工具长跨度完成任务，你需要强化学习。如果你想创造擅长总结或下一个 token 的模型，可能不需要强化学习。这是判断什么时候需要强化学习的好框架吗？

Federico

我觉得强化学习到处适用。即使是 Tab，我们也用了强化学习。我个人的理论——没有数据支撑——当你预训练模型，模型只是摄入人类知识的全部。比如你训练数学模型，模型学习 Stack Exchange 上的所有数学。一个没经过强化学习的模型，被呈现数学问题时，需要想自己是哪种人。是专家？还是学习的学生？

强化学习期间发生的一件事是我们调这个旋钮，让模型知道：嘿，你是专家，你得正确做事。这是我们锐化分布。强化学习有几个阶段。第一阶段模型很快变得非常好。然后第二阶段需要很多计算持续改进，模型开始推理，有这种 pattern。第一阶段我觉得我们就是调旋钮，告诉模型：嘿，你应该在这里正确做事。所以在少量计算时，强化学习也非常有用，只是让模型知道它得正确做事。

Dima

我同意。我们看这个 pattern 因为很多情况下我们帮助客户做强化学习微调。常规监督微调简化说是知识迁移，强化学习是锐化行为或你想从模型得到的特定品质。通常你两者都需要。即使你说的总结例子，强化学习可能很有用，因为有时候如果你想要特定风格的总结，很难想出好总结和坏总结的例子。很难精确描述。

但如果你用 LLM 作为评判者，你可以用非常精确的 rubric，提示模型说：这是我评估总结好坏的标准，扔进强化学习循环，让模型实验不同总结风格，找出你实际想要的。另一个 LLM 评估它是否匹配特定 rubric。这种 pattern 很多，不只是编码。

LLM 作为评判者

Host

Dima，我问你这个问题，因为 Federico 会拒绝回答。你提到几次 LLM 作为评判者。你认为最终公司会更成功有专家手检强化学习 rollout，手教练模型行为，还是 LLM 作为评判者、其他自动 rubric 能带我们到那里？

Dima

你不会真的让专家直接评判一般 rollout。那会是某种——实时强化学习如果是真实用户或者某种 RLHF 或 DPO。一般来说奖励越可验证越好，因为让你能扩展计算，得到更好结果。可验证意思是你怎么能自动生产它不用人？

如果是数学或编码，你能造非常确定性的东西，那是最好。LLM 作为评判者工作是因为生成器判别器区别——评判比创造容易，对人也一样。评判比创造容易。LLM 也有这个特性。

你可以精确造出你想用不同标准评判答案。你可以看到这种 pattern：你可能有非常复杂的多方面评估。如果你把多个方面扔给一个 LLM，它可能困惑怎么评判。你可以拆分：好，你基于风格评判，基于不同方面——造这些奖励。有些是确定性的，有些是 LLM 基础的，那就是引导你模型行为的。然后你开更多计算，看曲线上升。

关于专家，专家仍然需要，因为造这些任务和编码你想的产品体验，那才是关键。我们经历了软件 1.0、2.0、3.0。从直接造软件，到造训练数据。现在你实际上造评估规则。但这仍然非常重要。你需要看例子，看数据，看你的产品在哪里失败，怎么把模型推向正确行为。

两方面。第一，看前沿实验室，它们想造对一切都好的通用模型，需要覆盖所有不同任务，包在一个模型里，鼓励它泛化。这是一种 pattern，很有帮助。像 Composer 这种情况，你有你真实产品，这就是 Fireworks 的看法：如果你有真实产品，你应该对它做强化学习。最强的环境是你自己的产品。

因为那是你模型实际会用的地方。当然，如果前沿实验室，你会在所有产品上做。但如果你想为你的产品造最好的模型，专门化和定制它。你应该就用你的生产环境。当然你想适当隔离，不想模型对你的生产数据库捣乱。你想克隆它等等。有底层公司的一些通用基础设施工具让这更容易。

我们很早就发现这个，和很多客户一起工作。像 Cursor 的训练器在我们这边，其他一些客户，我们在我们的训练平台跑训练器。但环境，我们默认在客户侧跑，因为那是实际实现所在。你可以有同样的训练器设置，即使是 Fireworks 平台的一部分或在客户侧，调用真实生产环境，而不是试图包装它、组件化它。在托管平台上，因为那很难，引入差异。

Federico

强化学习环境实际是三个组件。一个是 harness——模型可以提交工具、工具被执行的地方。第二是某种操作系统——模型交互的实际世界、终端状态。然后是奖励组件——需要在最后检查工作是否正确完成。Harness 相当便携，你可以拿 harness 放很多不同环境。

关键的是操作系统。用普通容器复制这不太行。Cursor 我们实际造了整个虚拟机栈，可以很快启动虚拟机。而且必须超级爆发式，因为你能想象我们让系统：请现在给我十万个虚拟机。它得全都起来。
