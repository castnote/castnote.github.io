构建 Pi，以及为什么自我修改软件如此迷人

本期节目邀请 Mario Zechner（Pi 创建者）和 Armin Ronacher（Flask 创建者）探讨编码智能体的现状与未来。讨论涵盖了 Pi 的诞生故事——Mario 因不满现有智能体工具的复杂性和不稳定性，构建了一个极简但高度可扩展的编码智能体。核心话题包括：智能体如何因复杂性而自我削弱（添加的代码越多，context window 问题越严重）；AI 不感受代码库的痛苦，这导致缺乏重构动力；Armin 采访 30 多个工程团队的发现——PR 审查压力增大、代码质量下降、非工程师参与工程流程；MCP vs CLI 的权衡（MCP 适合企业认证，CLI 提供更好的可组合性）；以及「暗工厂」概念的批判——用大规模智能体军队生成软件可能带来垃圾代码洪流。两位嘉宾一致认为行业需要慢下来，把智能体用于自动化繁琐任务，而非追逐速度 KPI。

嘉宾背景与入行经历

Gergely Orosz

谢谢邀请我们。

谢谢。

Gergely Orosz

作为开场，Mario，你是怎么进入技术领域，最终开始做 AI 相关的东西的？

Mario Zechner

这是个很长的故事，我们有多少时间？我是 90 年代的孩子，96 年有了第一台电脑。触发点是我喜欢电脑游戏。我们家算是工薪阶层，买不起 Game Boy、NES、超级 NES 这些东西。但我有个叔叔有一台 Amiga 500，我每隔一天就去他那里玩游戏。

后来我父母跟我说，如果你去工作，攒钱就可以自己买一台电脑。实际上我爸会在正常工作之外去修车、去工地干活，这在欧洲很常见。两三年后他们说时机到了，带我到附近大城市的电脑店买了一台 486，就这样开始了。

我一直很喜欢游戏，这也引导我接触了图形编程。后来在大学读书时很幸运地找到了一份工作，在一个应用科学组织做 NLP、机器学习相关工作，把研究成果应用到工业场景里。那就是我学到机器学习基础的地方，当时深度学习还没成为主流。

2010 年左右我离开了这个领域，去旧金山加入了一家创业公司。后来又跟两个朋友在瑞典做了另一家创业公司，做的是 Java 到 iOS 的提前编译器，公司后来被卖掉了。之后我有了更多时间，但我一直关注机器学习的进展，因为确实很有意思。然后 GPT 出现了，就这样。

Gergely Orosz

Armin，你的起点是怎样的？

Armin Ronacher

我的起点肯定不是工薪阶层。我父母经营一家建筑设计事务所，他们很早就采用电脑做 CAD 绘图，所以我最早的电脑是他们淘汰下来的旧机器。我第一台电脑虽然是更年轻的时候，但却是 386。很抱歉你听到这个。

基本上我所有的电脑都没法正常运行游戏，因为他们用的是 Windows NT，当时什么都不支持。唯一的办法是在启动到 Windows 95 或 Windows 3.11 之前，先启动到 DOS，这样才能玩一些很老的 DOS 游戏，那时候已经有更好的东西了。正因为这样，我开始折腾 Quick Basic 和 Turbo Pascal，买了好多相关的书，这就是我学习的起点。

说实话我一开始真的很不擅长，但我觉得很有趣。如果你持续做下去，就会变好。2002 或 2003 年时我经常用 Delphi，那是 Turbo Pascal 的可视化版本。有人给我展示了 Linux，但 Delphi 在 Linux 上用不了，于是我找到了 Python。

Ubuntu 2004 年发布，它是风投支持的项目，但创建了本地社区。我跟朋友一起创办了德国 Ubuntu 协会，运营了一个叫 Ubuntu Users 的在线社区四五年的时间。因为 Ubuntu 很流行，社区增长很快，然后出现了扩展性问题，这就是我进入 Web 开发的原因。

为了构建这个，我想做一个模板引擎、一个 Web 库，最终我把这些东西打包在一起，做了 Flask 框架。它很流行，现在也还是很多人喜欢用的东西。2013、2014 年我在伦敦做了几年游戏开发，之后回到开源领域，在 Sentry 工作了 10 年，去年四月离开去尝试其他事情。

你们两个人都来自奥地利，实际上你们现在也住在奥地利对吧？你之前在 Sentry 工作，也做过游戏。还有第三个人，Peter Steinberger，也来自奥地利，他刚刚也在这个播客上。你们是在哪里认识的？我最近看到你们在 OpenClaw 和 Pi 开始之前一起出去玩的照片，三个人在实验、玩 AI。

Mario Zechner

我觉得我们两个是在网上认识的，Reddit 上？

Armin Ronacher

实际上我大学时见过你一次，但那时候你没认出我，我当时还没出名。但我们算是抽象地在网上认识了。

Mario Zechner

后来我们在维也纳见面，我们在网上互相大吼，但是以一种很可爱的方式，不怎么对立的方式。虽然我们在生活的很多领域想法不一样，但交流是有礼貌的。

Peter 是六度 Steinberger 的概念。我在我住的城市有一个办公室，一个公司给我免费办公空间，条件是给 CEO 当导师。这个公司和 Peter 的公司 PSPDFKit 有业务往来。Peter 来 Graz 的办公室，那是我们第一次见面。同一年我们在伊斯坦布尔的一个会议上也碰到了，整晚待在一起，那就是一切开始的地方。

从怀疑 AI 到拥抱 AI

Gergely Orosz

你们两位是怎么从最初对 AI 工具持怀疑态度，到最终认为这些东西真的很有意思的？在 2022 年时，你们都已经有十多年构建复杂软件的经验了。你们的第一反应是什么？然后是怎么转变到觉得这东西确实有意思的？

Mario Zechner

对我来说，2022 年时 Copilot 已经出来了，比 GPT 更早。我之前创业公司的东西让我认识 Nat Friedman 和 Miguel de Icaza，因为他们收购了我们做 Java 编译器的公司。Nat 后来去了 GitHub。2022 年他私信问我要不要 GitHub Copilot 的早期访问权限。

我当时说我不怎么在乎，觉得这东西不会有什么前途。他说这是未来，你得试试。所以我试了一下，确实很糟糕。但当 GPT 出现，尤其是他们开始提供 API 访问后，我做了很多项目，研究什么能做什么不能做，不一定是在编程领域。

当他们有了 tool calling 或 function calling 功能后，变得很有意思。但要到 2024 年 10 月左右才真正有用。那是编码智能体开始变得有意思的时候。2025 年，Claude Code 团队发布了 Claude Code，引入了 agentic search，基本上就是给智能体一个方式去遍历你的文件系统、读取所有文件。

这彻底改变了局面。之前像 Cursor 用索引、AST 相关的东西，这些都不重要了。这是给智能体访问你的文件的权限。对我来说，这才是关键点。

Armin Ronacher

我的路径差不多。Copilot 出来更早一些，GitHub 有一个维护者计划，我有早期访问权限。我当时感觉到 Copilot 会很有意思，但不是现在这样的方式。

我在开源领域很长时间，他们用开源数据做训练，这会很有争议。我没想到它会有生产力。我记得当时在试探它，比如问它会不会吐出 GPL 代码。有一次我让它吐出了 Carmack 的逆平方根函数，那个函数有很特别的名字所以很容易被召回。

我发现如果用某种方式 tab，它会继续输出许可证文本，但完全是错的。它来自 GPL 的 Doom 源码，但它却标注了一个随机的 MIT 许可证。当时我发了一条推文说 Copilot 做错了事情，那条推文变得很火。

通过这条推文和其他事情，我开始意识到实验室里确实有真正的 AI 进展。我没学过 AI，只知道有 AI 寒冬然后什么都没发生。但有些公司的 CEO 相信这会有发展。我开始用 API 尝试各种东西，比如能不能做 bug 修复，开始变得有兴趣，但还感觉不到世界会改变。

关于吐出开源代码这件事，我改变了立场。我很多年的立场是希望大家分享东西，人类进步是建立在彼此之上。我喜欢这种海盗式的分享知识的做法。我的理想版本是版权几乎不存在或非常有限。所以我不在乎它吐出 GPL 代码且不标注来源。如果结果是版权被完全摧毁，我也没问题。

但有趣的是它一开始确实创造了许可证违规的问题。我想看看会涌现出什么样的混乱。到目前为止，涌现出的是我们对版权系统的一些假设被挑战了，我们现在都在忽略这些问题，因为我们想先制造混乱然后再重新监管。

采访 30 多个工程团队的经验

Gergely Orosz

说到今天，你最近作为新创业公司的一部分，跟大约 30 个不同的工程团队聊了他们如何在公司内部、团队内部使用智能体。从大公司到创业公司，你学到了什么？

Armin Ronacher

很多发现其实并不意外，就是当人们有假期时会花更多时间尝试这些工具。

Gergely Orosz

你跟了很多人聊对吧，从 Meta 这样的创业公司到各种不同的公司？

Armin Ronacher

各种各样的人，从欧洲的老牌企业比如 Siemens，到关键领域的公司。当人们有假期时他们才会真正去尝试，当技术负责人来说你现在得用 Cursor、用 Claude Code，其实你不真正理解。你需要花两到三周时间才能真正理解。

我离开公司后有大量空闲时间，从四月到十月我可以深入这些东西。我觉得像猫薄荷一样让人上瘾。但在公司内部，感恩节、夏天、圣诞节期间，人们会尝试，而且那时候经常有免费额度。圣诞节之后，超过一半的公司里使用量爆炸了。

使用量爆炸的方式就是质量下降了。不是因为人们想写更差的代码，而是因为这需要努力才能保持质量。去年夏天我们在创业公司生态里已经看到了。你看 YC 的创业公司，很多人把代码放在 GitHub 上，你能看到 plan.md 文件、所有代码都标注是 Claude 写的，那种 vibe coding 的方式。

工程团队和公司现在对此的反应很有意思。挑战之一是审查 PR 变得更难了。PR 变得越来越大、越来越频繁，而且心理负担更重。PR 里的很多代码不是工程师会写的方式。作为工程师，你会想为未来的自己负责，但智能体不在乎。

我讲一个故事。我在 Xbox One 发布时做了一个游戏，那是个固定发布日期的项目。我参与过 Halo Master Chief Collection，有个匹配组件。那是所有人上阵的情况，我们要去清理人工制造的垃圾代码。系统有太多状态，我们叫它 emergent state machine，16 个布尔变量组合成几何级数增长的可能状态。

这就是智能体代码的感觉：理论上应该只有清晰定义的几个状态，但现实中它会做各种恢复处理，比如配置加载失败就捕获异常然后加载默认配置。这让代码变得更复杂，因为它不是正确失败而是在恢复并进入更多失败状态。这让代码更难工作，因为你也没办法让智能体反思，它会说这可能是需要的。

Mario Zechner

我觉得比你描述的人类制造的复杂系统还要糟糕。因为智能体有时会有令人惊叹的时刻，输出完美简洁的代码，就是你需要的代码。你作为驾驶员工程师看着觉得太棒了，我可以不用管了。

两分钟后，另一个智能体在这个窗口里运行，输出最糟糕的垃圾。但你可能注意不到，因为你现在陷入了自动化偏见，觉得你的智能体工作得很好。

Gergely Orosz

你觉得这可能是人类的偏见吗？因为入职新工程师、新毕业生，你会审查他们的代码。如果代码很差，你会仔细审查下一个直到他们写的代码跟你一样。这需要六个月或一年。然后你可以信任这个人。

Mario Zechner

但智能体没有这种学习。智能体不会学习，你可以在 system prompt 里放很多东西、构建记忆系统，但那不是人类的学习。人类会学习，而且人类会感受痛苦。我认为这是人类的决定性特征。

回到你说的事情，如果痛苦太大，人类就会被激励去解决痛苦的根源。在代码库中，根源通常是糟糕的接口、糟糕的复杂性，你想消除它因为你没法维护那个系统了。

Gergely Orosz

这就是为什么资深工程师总是有需求的原因吧？CEO 认为资深工程师就是把事情搞定，但实际上，有效的资深工程师都有战伤，他们被烧伤过，他们感受过痛苦，他们看到过技术债务螺旋的结果。所以他们做出那些决策来避免问题。

Mario Zechner

我个人认为，一个好的工程师是经常说不的工程师。说我不需要这个、不需要那个。这能保持复杂性低。如果你用智能体，恰恰相反，你说是的我想要这个那个还有这个，因为我不需要自己打字、不需要思考。我只需要给机器一个提示，它就输出一个看起来差不多是我想要的东西。这就开始了所有问题。

Armin Ronacher

好的工程是知道你要做出的权衡。有时候正确的解决方案实际上是你大学里学到的你不应该做的那个。Cal Henderson 曾经说过，先做最笨的解决方案，直到它不行了再换。因为如果你一开始就做正确的、完美的方案，你创造的复杂性会在规模上杀死你。

工程师通过战伤学到这些，但如果你没有那些经历，很难正确地论证。这个学习过程给你权威去说服其他工程师应该这样做。但智能体现在给你世界知识访问。资深工程师说不之后，48 小时后初级工程师说我问过智能体，我已经有证据为什么不应该这样做。

现在产生了新的压力，这有点像人们带着 ChatGPT 打印输出去找医生说机器说了你得这样做。

Gergely Orosz

基于你观察到的，资深工程师可能更难对产品经理或初级工程师说不？

Mario Zechner

更糟糕，因为产品经理现在会发送 pull request 并且自动合并。

Armin Ronacher

非工程师参与工程流程现在是个事情。

Mario Zechner

问问 Gergely 这事怎么样。

Gergely Orosz

Armin，你的体验是什么？是你自己的公司还是跟别人聊天？

Armin Ronacher

我们很小，我的联合创始人有时会给网站发博客文章。我跟别人聊过，营销团队突然在网站上做事，销售团队创建越来越复杂的销售演示，这些东西会落到 GitHub org 上。最有趣的一个例子是销售演示构建了一个不存在的产品功能，但没人发现。这是全新的东西。

Mario Zechner

这很赋能。如果你的整个组织里每个人都能参与某种形式的软件创建，以前做不到这些。设计师能在 Figma 里设计东西，但可能没法把它变成可点击的演示。产品经理想尝试一个功能但又不想浪费工程师时间，现在可以做了。

问题是人们现在太关注于所有人都能做所有事情了，忘记了你还需要流程来控制这些东西。

Armin Ronacher

整合部分是最难的。Peter 提出了 prompt request 的想法，我越来越喜欢这个想法。一旦你演示完，我不再需要你的代码。

Gergely Orosz

回顾一下，prompt request 是他说他不喜欢收到 pull request，宁愿看到提示，因为他会运行提示或调整它，生成符合他自己风格的代码。

Armin Ronacher

对我来说，不只是我想看到提示，我想知道它要做什么。实际上很多时候你并不完全知道你一开始想做什么。创造的行为本身会澄清你真正想做什么。这部分很有价值。

很多时候方案和代码不是有足够经验的工程师会做的。所以我不是要你的提示以便重新让智能体生成稍好一点的版本。而是我们知道要构建什么之后，可能我更快从头开始。

Mario Zechner

我其实不完全同意 Peter 的我只需要你的提示。我确实觉得看到糟糕的实现有价值。如果收到一个 pull request，大部分 Pi 仓库的 pull request 是智能体写的没有太多人工干预。我知道这会是垃圾。但这是有价值的垃圾，因为有人至少花了一点时间指示智能体创建这个 pull request。

我可以看到他们想要的东西的糟糕实现是什么样，我不需要浪费自己的时间去尝试。有人已经试过了，天真的智能体版本。这节省我的时间。不是说我喜欢智能体的 pull request，它们很糟糕我现在会自动关闭它们，但它们有价值。不只是提示。

这是指数级的，一切的速度。当然最终是 sigmoid 因为热力学限制。但我觉得我们会比之前的循环更早发现这是个坏主意。这是好消息。

Armin Ronacher

我觉得有趣的是，我读了一篇关于英国工业革命如何改变纺织业的精彩文章。核心观点是每次管道前端被优化，就会创造下游的激励去制造某些东西。一开始如果你能更快织布，最终你需要能以更快速度纺纱的纱线。最终整个链条的瓶颈都在移动。

最终最大的瓶颈变成了我们工程领域下一个会遇到的瓶颈：以前你做一件衬衫，如果不喜欢，去找做的人他会帮你修。现在如果衬衫不好，没人关心过程中谁毁了衬衫。你直接换一件新的。责任从链条里的任何人转移到了整个工厂作为一个整体不再承担责任，因为我们把整个事情商品化到不需要这样做。

如果你用工程的角度，运营公司和服务的一个重要部分是可靠地运营。所以你有事故复盘来搞清楚过程中哪里出错了。现在机器以 10 倍速度产出东西，责任机制没法以同样方式扩展，因为机器不能承担责任。

我不确定有没有一个未来可以抽象掉人类失败，让我们运营工程的方式不再关心谁签批了 pull request。我们自动化创建衬衫的方式。我还没看到。

Mario Zechner

复杂性是智能体最大的敌人

Gergely Orosz

Mario Zechner

很简单。如果我有 600k 行代码，我的智能体最好能在大约 200k token 的 context window 里有效工作。它能看多少代码？三分之一吧。如果你能把任务相关的所有代码放进 context window，大概没问题。
